SubCAD Natural Feature Specification Ideas

SubCAD should let users describe design intent the way machinists, sheet-metal designers, and mechanical drawings usually specify it: offsets from edges, symmetry, patterns, constraints, references, and manufacturing intent. Raw absolute coordinates should remain available, but they should not be the only practical way to place geometry.

edge offsets symmetry patterns constraints named references manufacturing intent

This is an idea catalog for later evaluation. It intentionally mixes near-term language sugar with deeper modeling features so we can decide what belongs in the first implementation pass.

Motivating Example

For a rectangular metal plate with four corner holes, a designer usually says something like "M6 clearance holes, 12 mm from each edge" or "four holes on the corners, inset 10 mm." They rarely think in absolute XY coordinates unless the part is already fully dimensioned.

plate 120 x 80 x 3 corner_holes diameter 6 inset 10 // Equivalent intent: // holes at: // left + 10, bottom + 10 // right - 10, bottom + 10 // left + 10, top - 10 // right - 10, top - 10

Feature Placement Ideas

1. Edge-Based Offsets

Place features by distance from one or more named edges instead of absolute coordinates.

hole diameter 5 from left 12 from top 12

2. Corner Insets

Treat corners as first-class anchors and allow a uniform or per-axis inset.

corner_holes diameter 6 inset 10

3. Centerline References

Place features on horizontal, vertical, or named centerlines, then offset along the line.

slot on vertical_center from top 20 length 30 width 6

4. Symmetric Pairs

Define one feature and mirror it across a centerline, plane, edge midpoint, or named axis.

hole diameter 4 from left 15 from top 18 mirror across vertical_center

5. Rectangular Patterns

Create grids from count, spacing, and direction, with optional centering or boundary margins.

hole diameter 3 pattern rows 2 cols 4 spacing 20 centered

6. Linear Patterns Between References

Distribute features evenly between two edges, points, holes, or named construction references.

5 holes diameter 4 between left+12 and right-12 on horizontal_center

7. Equal Spacing Constraints

Let the system solve spacing from margins and count rather than requiring every location.

holes count 6 diameter 5 margin 15 equal_spacing along width

8. Relative Feature Placement

Position a new feature relative to an existing feature, such as another hole, slot, tab, or bend.

hole diameter 4 right_of mounting_slot by 25 aligned_y

9. Named Anchors

Allow designers to name construction points or regions and reuse them across later operations.

anchor motor_mount at center offset x 20 hole diameter 6 at motor_mount

10. Feature Groups

Group related geometry so transformations, mirrors, suppression, and parameter changes apply together.

group handle_mount: 2 holes diameter 5 spacing 32 centered at top inset 18

11. Boundary-Aware Placement

Define features in terms of the current part boundary, not just the original rectangle. This matters after chamfers, notches, flanges, tabs, and cutouts.

hole diameter 5 inset 10 from nearest_outer_edges

12. Region-Based Placement

Attach features to logical regions such as flange face, web, tab face, front panel, or bracket leg.

on front_flange: corner_holes diameter 4 inset 8

Constraint and Intent Ideas

Idea What the user says What SubCAD should infer
Keep clearance "Put the hole 2x diameter from the edge." Compute edge distance from the chosen hole diameter and preserve it when diameter changes.
Maintain symmetry "Make these holes symmetric about the center." Store the mirror relationship, not just the copied coordinates.
Align features "Align this slot with those holes." Constrain X, Y, or centerline alignment across feature centers.
Tangent or flush placement "Put the slot flush with the inside wall." Resolve the feature boundary against another edge or face, accounting for feature size.
Minimum material rules "Keep at least 8 mm of metal around all holes." Validate edge distances and feature-to-feature distances against a rule.
Driven dimensions "Hole spacing is plate width minus 24." Maintain formulas and update placement when parent dimensions change.
Manufacturing standard "Use standard M5 clearance holes." Map the user intent to a diameter, tolerance, and optional annotation.
Preferred orientation "Slot runs along the long direction." Choose orientation from part proportions or the selected region.

Natural Operations Beyond Holes

Slots and Cutouts

Support edge offsets, centerline placement, symmetric pairs, and clearance rules for slots, rectangular cutouts, keyholes, louvers, and vents.

Bends and Flanges

Specify bend lines by offset from an edge, flange height, inside radius, and whether the material grows inward, outward, or about the neutral line.

Tabs and Notches

Attach tabs to edges with count, width, spacing, clearance, and matching notch generation on mating parts.

Mounting Patterns

Provide named patterns such as motor mount, fan mount, DIN rail, VESA, bearing block, standoff set, hinge, handle, and connector panel.

Corner Treatments

Apply chamfers, radii, dogbones, bend reliefs, or corner clearances using corner names and manufacturing rules.

Fastener Stacks

Capture through-hole, counterbore, countersink, tapped hole, captive nut, insert, and clearance washer intent as one semantic operation.

Reference Vocabulary

A useful natural layer needs a stable vocabulary of references that can be resolved by the compiler before geometry generation.

Reference type Examples Notes
Part edges left, right, top, bottom, front, back, nearest edge Needs a consistent coordinate convention per sketch or face.
Part centers center, vertical centerline, horizontal centerline, face center Should update when part dimensions change.
Corners top-left, top-right, bottom-left, bottom-right Can expand into paired edge offsets.
Faces and regions front panel, flange, tab, web, base plate, side wall Important for bent sheet-metal and assemblies.
Existing features hole center, slot centerline, cutout edge, bend line Enables relative placement and alignment.
Construction geometry datum point, named axis, sketch line, construction rectangle Useful when natural references are not enough.

Possible Compiler Model

The natural language or declarative layer can compile into explicit SubCAD operations in stages. This keeps the final geometry deterministic while preserving design intent upstream.

  1. Parse the user's feature intent into semantic operations such as corner hole pattern or slot on centerline.
  2. Resolve references such as edges, corners, faces, centerlines, named anchors, and existing features.
  3. Evaluate formulas and manufacturing standards, including units, hole tables, clearances, and minimum material rules.
  4. Expand high-level operations into explicit lower-level geometry operations with computed coordinates.
  5. Retain the source intent so later edits can change "inset 10" instead of editing four independent coordinates.
  6. Validate the result and report conflicts such as impossible spacing, edge violations, overlaps, or ambiguous references.

Priority Candidates

Priority Candidate Why it matters first
1 Edge offsets and corner insets Solves the common plate-hole case and maps cleanly to current coordinate placement.
2 Symmetric pairs and four-corner patterns Reduces repeated coordinates and captures real design intent.
3 Centerline and equal-spacing patterns Useful for slots, vents, mounting rows, and scalable part dimensions.
4 Named anchors and feature groups Prepares the system for more complex assemblies and reusable design snippets.
5 Manufacturing rules and standards Adds trust, but should come after the placement model is stable.

First Implementation Pass

The initial SubCAD implementation now covers natural X/Y placement for rectangular stock by compiling edge, corner, centerline, symmetry, and rectangular pattern intent into the existing explicit coordinate-based operations.

Implemented API Supported intent Compiles to
resolve_xy(...) Resolve one X reference and one Y reference from edge offsets, centerlines, or explicit coordinates. (cx, cy)
drill_at(...) Drill by saying things like from_left=10, from_top=12. drill(..., cx=..., cy=...)
corner_holes(...) Four-corner or selected-corner hole patterns using uniform or per-axis insets. Repeated drill(...) calls
drill_symmetric(...) Mirror a hole across the X centerline, Y centerline, or both. Repeated drill(...) calls
drill_pattern(...) Centered rectangular row/column hole grids from count and spacing. Repeated drill(...) calls
pocket_at(...), circular_pocket_at(...), slot_at(...) Use the same natural X/Y references for pockets and slots. Existing pocket, circular pocket, and slot operations

Machine-Aware Process Context

SubCAD operations should be validated against the actual manufacturing context, not only against geometric possibility. A process plan now needs to say which machine it is intended for, because a valid feature on one machine may be impossible, unsafe, or require extra setups on another.

Missing context Why it matters How to incorporate it
Machine Defines axes, travels, spindle limits, rotary capability, and physical envelope. Add with_machine(...) / on_machine(...) and validate the plan against machine metadata.
Controller/post Even when the machine can move, the controller may not support a cycle, macro, probing call, or rotary mode. Keep controller capability separate from machine hardware and validate before postprocessing.
Setup strategy Side-face work on a 3-axis mill is possible only if the part is re-fixtured or indexed. Require named setups, face selectors, work offsets, and setup-specific fixture data.
Workholding Clamps, vises, soft jaws, and tombstones decide whether a toolpath is safe. Attach fixture specs, clamping zones, clearance zones, and stock fit checks to the plan.
Tool magazine A theoretically suitable tool may not be in the shop, may not fit the holder, or may exceed stickout limits. Validate tool availability, holder interface, tool reach, flute length, and magazine capacity.
Coolant/probing/inspection Deep holes, hard materials, and tight datums often require coolant mode, probing, or in-process inspection. Represent coolant requirements, probe cycles, datum verification, and inspection checkpoints.
Tolerance and surface finish The same geometry can be rough-machined or tolerance-machined with very different machines and tools. Attach tolerance, finish, datum, and verification intent to operations or feature groups.

The machine files now carry an explicit capability schema. SubCAD should prefer these declared capabilities over guesses from axis count alone.

{ "capabilities": { "schema_version": "subcad.machine_capabilities.v1", "processes": ["mill", "indexed_3axis", "5axis"], "operation_families": ["drill", "pocket", "slot", "surface_mill"], "controller_cycles": ["G81", "G83", "G84", "TCP"], "coolant": ["flood", "through_spindle"], "probing": true, "tool_interface": "HSK-A63", "tool_capacity": 30 } }

Validation now checks operation family, controller cycle, coolant mode, probe requirement, tool interface, tool capacity, setup access, controller/post support, tooling availability, inspection requirements, and collision gates in addition to axes, travels, and spindle/feed limits.

from src.subcad.validation import validate_all from src.subcad import load_machine_context machine = load_machine_context("machines/generic_3axis.json").to_dict() plan = { "schema_version": "subcad.shop_floor.v1", "material": "aluminum_6061", "stock_dimensions": {"length": 100, "width": 80, "height": 20}, "machine": machine, "operations": [ {"operation": "surface_mill", "process": "mill"}, {"operation": "drill", "controller_cycle": "G234"}, {"operation": "drill", "coolant": "through_spindle"}, {"operation": "drill", "requires_probe": True}, {"operation": "drill", "tool": {"interface": "HSK-A63"}} ] } report = validate_all(plan, {}, structured=True) print([issue.code for issue in report.issues]) # machine_operation_unsupported # machine_controller_cycle_unsupported # machine_coolant_unsupported # machine_probe_unsupported # machine_tool_interface_mismatch

Second Implementation Pass

The process context layer now makes manufacturing assumptions explicit. A part can carry machine, controller, setup intent, tooling, quality, and collision-policy information while still compiling natural feature placement into the same lower-level X/Y operations.

Implemented API Supported intent Validation examples
with_controller(...) Attach controller/post capabilities such as cycles, coolant codes, rotary modes, probing, macros, and work offsets. Unsupported G234, unavailable probing, rigid tapping, macro, rotary mode, or work offset.
with_setup_intent(...) Describe workholding, datum, orientation, primary face, accessible faces, setup count, and work offset. Operation face is not accessible, or requires another setup/reorientation.
with_tooling(...) Attach tool assemblies and magazine slots/capacity/interface. Missing tool, wrong interface, exceeded magazine capacity, RPM/feed above tool limit, or through-tool coolant unavailable.
with_quality(...) Attach tolerances, inspection methods, surface finish, critical features, and process capability. Probe/CMM requirement without machine/controller support, missing inspection operation, or tolerance below declared process capability.
with_collision_policy(...) / require_collision_check(...) Require toolpath and holder/fixture collision readiness before shop-floor handoff. Missing toolpath, missing holder envelope metadata, or missing fixture context.
with_process_context(...) Attach all of the above in one fluent call. Runs the combined validation pipeline through validate_shop_floor(...).

Example: a natural edge-offset hole can carry the full process context and still match the explicit coordinate version.

from src.subcad import Stock from src.subcad.tooling_context import load_tooling_context tooling = load_tooling_context( "tooling/sample_tooling.json", "tooling/sample_magazine.json", ) setup_intent = { "setup_id": "op1_top", "workholding": {"name": "kurt_dx6_vise", "fixture_type": "vise"}, "part_orientation": {"primary_face": "top"}, "accessible_faces": ["top"], "operation_side": "top", "work_offset": "G54", "setup_count": 1, } quality = { "tolerances": {"default": {"linear_mm": 0.1, "methods": ["caliper"]}}, "inspection": {"required": False, "methods": ["caliper"]}, "collision_policy": {"enabled": False, "required": False}, } natural = ( Stock.rectangular(120, 80, 6) .with_process_context( machine="generic_3axis", controller="generic_3axis_debug", setup_intent=setup_intent, tooling=tooling, quality=quality, ) .drill_at(6, depth=5, from_left=10, from_top=12, spot_drill=False) ) explicit = ( Stock.rectangular(120, 80, 6) .with_process_context( machine="generic_3axis", controller="generic_3axis_debug", setup_intent=setup_intent, tooling=tooling, quality=quality, ) .drill(6, depth=5, cx=-50, cy=28, spot_drill=False) ) assert natural.process_plan()["operations"][0]["position"] == [-50.0, 28.0] assert explicit.process_plan()["operations"][0]["position"] == [-50, 28] natural.validate_shop_floor(raise_on_error=True)

Example: the same validation pipeline catches a plan that asks for unsupported controller, setup, tooling, quality, and collision behavior.

from src.subcad.validation import validate_all bad_plan = { "schema_version": "subcad.shop_floor.v1", "material": "aluminum_6061", "stock_dimensions": {"length": 120, "width": 80, "height": 6}, "machine": natural.process_plan()["machine"], "controller": natural.process_plan()["controller"], "setup_intent": setup_intent, "tooling": natural.process_plan()["tooling"], "quality": { "inspection": { "required": True, "methods": ["on_machine_probe"], "requires_probe": True, }, "collision_policy": {"enabled": True, "required": True}, }, "collision_policy": {"enabled": True, "required": True}, "operations": [{ "operation": "thread_mill", "tool_type": "thread_mill", "tool_diameter_mm": 8.0, "position": [0, 0], "face_selector": "<Z", "controller_cycle": "G234", "coolant": "through_spindle", }], } report = validate_all(bad_plan, bad_plan["stock_dimensions"], structured=True) print(sorted({issue.code for issue in report.issues})) # collision_toolpath_missing # controller_coolant_unsupported # controller_cycle_unsupported # quality_probe_unsupported # setup_face_not_accessible # tooling_tool_missing

Open Questions

  • Should this be a formal DSL, a natural-language compiler layer, or both?
  • How much design intent should be retained after compilation into geometry?
  • What is the minimum reference vocabulary needed for the first useful version?
  • How should ambiguous phrases like "top" or "front" be handled across 2D sketches, 3D parts, and bent sheet metal?
  • Should standard patterns live as built-ins, libraries, or user-defined reusable templates?
  • How should the UI expose these features: text, forms, handles, selectable references, or a hybrid?
