SubCAD Operation Video Simulation Plan

Status: V1 foundation partially implemented.

This plan documents the next SubCAD direction: time-based, realistic operation simulation and video output. The goal is to move beyond final STEP geometry and static turntable renders into a view of how the raw stock changes during each manufacturing operation.

Goal

Given a SubCAD program and process plan, generate an interactive and exportable operation video that shows:

  • The original stock, fixture/workholding, machine, tool, and target context.
  • Material removal or deformation over process time, not just the final part.
  • Half-finished states when the timeline is paused or scrubbed.
  • Tool changes, setup changes, and machine-to-machine transitions.
  • Captions for operations, tools, setups, and process transitions.

The reference example is a 100 x 50 x 50 mm aluminum block where a pocket is machined and edges are chamfered. At 50 percent of the pocket operation, the viewer should show a semi-finished pocket, not the final pocket.

Chosen V1 Direction

These defaults are chosen for the first implementation discussion:

  • Scope: milling and turning first.
  • Output: interactive timeline plus saved video export.
  • Fidelity: hybrid realistic.

Hybrid realistic means real geometric stock updates where the repository already has or can reasonably extend a simulation engine, plus believable tool, fixture, machine, and process animation around it. Processes without reliable material physics in V1 should be clearly labeled as visual approximations.

Implemented V1 Foundation

The first implementation slice is now present:

  • Stock.simulation_timeline(...) returns subcad.simulation_timeline.v1.
  • Stock.export_simulation_package(...) writes simulation_timeline.json, simulation_states/*.stl, and video_manifest.json.
  • The timeline uses seconds, process-plan operations, neutral toolpath moves, and estimated operation time.
  • Milling-style material removal can generate mid-operation tri-dexel stock snapshots for scrubbing and playback.
  • Stock.simulate() now starts from original stock replay instead of the final shape.
  • web/session-viewer.js loads optional simulation_timeline assets and uses the nearest stock keyframe during playback/scrubbing.

This is not yet the full factory-video system. Browser recording, headless video export, named machine visuals, turning profile simulation, and non-milling process adapters remain future phases.

Current Repository Baseline

SubCAD already has several useful building blocks:

  • Fluent Stock API with process-plan JSON and setup-sheet export.
  • Neutral toolpaths with rapid/feed/plunge/retract/dwell/arc moves.
  • Tool, holder, fixture, setup, work-offset, quality, and economics metadata.
  • Tri-dexel material-removal structures under src/simulation.
  • FiveAxisSimulator, material removal, tool geometry, machine models, and browser visualization packages.
  • web/visualization.html and web/session-viewer.js for Three.js review of stock states, fixtures, tools, toolpaths, and operation metadata.
  • Browser WebGPU prototype files under web/sim and web/shaders.

Latest local checks before this plan was written:

  • python test_sim_bridge.py passed 51/51.
  • python test_subcad_phase2_toolpaths.py passed 154/154.
  • python test_subcad_visualization.py passed 23/23.

Important Gap To Fix First

The current Stock.simulate() bridge initializes tri-dexel simulation from self.to_mesh(). On a completed SubCAD chain, that mesh is the final part, not the original stock. Operation-video simulation must initialize from the original stock state and then replay operations forward.

Implementation should use one of these sources, in priority order:

  • The initial entry in _state_history if present.
  • A new explicit initial-stock mesh captured when stock is constructed.
  • A reconstructed stock mesh from process_plan.stock_dimensions only as a fallback.

This gap has been fixed for the current simulation bridge and operation-video timeline path. Without this boundary, a video simulation can look plausible while being physically wrong.

CadQuery Boundary

Partial operation keyframes are not animated STEP/B-Rep states and should not be generated by rerunning CadQuery booleans frame by frame. CadQuery/SubCAD remains the final design authority for the finished part and STEP export.

The operation-video layer is a separate process replay model:

  • Start from the original stock mesh.
  • Replay process-plan/toolpath/tool metadata over estimated operation time.
  • Subtract swept tool volumes in the tri-dexel simulation engine where supported.
  • Export visual mesh snapshots for the browser viewer.

Those snapshots are for visualization and review. They are not a replacement for the final STEP file or a production CAM verifier.

Proposed Interfaces

Stock.simulation_timeline(...) -> dict is implemented.

Recommended parameters:

  • fps=30
  • playback_rate=1.0
  • resolution_mm=0.5
  • keyframe_interval_sec=0.25
  • target=None
  • include_machine=True
  • include_fixtures=True
  • include_process_effects=True

Return schema: subcad.simulation_timeline.v1.

Core fields:

  • units
  • source_plan
  • timeline
  • operations
  • tools
  • machines
  • fixtures
  • stock_keyframes
  • captions
  • fidelity
  • performance

Stock.export_simulation_package(output_dir, ...) -> dict is implemented.

This should extend the current visualization package with:

  • simulation_timeline.json
  • simulation_states/*.stl
  • video_manifest.json
  • machine, fixture, tool, and effect assets
  • a viewer entrypoint for playback and recording

Add a command-line export path:

python -m src.subcad.video_export path\to\scene.json --out output.webm

Browser-native export should default to WebM. MP4 can be added later through ffmpeg when available.

Timeline Model

The timeline compiler should convert the process plan into display segments:

  • setup_transition: move part to a new machine or setup.
  • tool_change: old tool retracts/exits, new tool enters.
  • rapid_move: non-cutting positioning.
  • cutting_move: material-removal motion.
  • forming_move: press brake or deformation motion.
  • process_move: welding, EDM, laser, grinding, or other visual process.
  • inspection_or_pause: optional labeled pauses.

Segment timing should use authored data in this order:

  • toolpath_summary.estimated_time_min
  • pass_plan.estimated_time_min
  • operation economics timing
  • polyline length divided by feed rate
  • conservative default fallback

Each segment should include:

  • start and end timestamps
  • operation sequence number
  • machine/setup/tool IDs
  • tool pose path
  • cutting or non-cutting flag
  • stock keyframe references
  • caption text
  • fidelity label

Simulation Engines

Milling

Use the existing tri-dexel material-removal engine as the main source of truth. Replay neutral toolpaths move by move and generate stock keyframes at a fixed time interval or after meaningful material changes.

Default display should show:

  • stock mesh changing over time
  • fixture and clamps
  • spindle, holder, cutter, and optional shank
  • toolpath trail
  • current operation and pass
  • target ghost mesh when useful

Turning

Add a lathe-specific fast path rather than forcing turning into milling logic. Represent stock as an axisymmetric radius profile along the spindle axis, then revolve it to a display mesh for keyframes.

Default display should show:

  • rotating chuck and stock
  • stationary or moving turning tool
  • faced ends, OD turning, ID boring, grooves, and threads as progressive profile edits

Other Processes

Support the same timeline schema in V1, but use explicit fidelity labels:

  • Laser cutting: 2D kerf progression on sheet stock, pierce dwell, glow trail.
  • Press brake: punch/die motion and bend-angle interpolation.
  • EDM: electrode/wire path, spark effect, progressive removal approximation.
  • Surface grinder: wheel pass animation and shallow material-removal bands.
  • Welding: torch path, bead deposition visual, heat tint, no structural physics.

These processes should not pretend to be fully validated physics in V1.

Browser Viewer And Video Export

Extend the current session viewer to support:

  • time scrubber in seconds and percent
  • operation list synchronized with timeline
  • playback speed control
  • current tool, operation, setup, and pass display
  • stock state interpolation or keyframe switching
  • captions during machine transitions and tool changes
  • record/export button

Video export path:

  • Render frames in the browser viewer.
  • Capture the canvas through captureStream.
  • Encode with MediaRecorder to WebM.
  • Use fixed camera and deterministic playback for repeatable output.
  • Keep the generated video duration based on estimated process time, with optional speed-up for review videos.

Target performance for the reference pocket/chamfer case:

  • Generate video at least 10x faster than output video duration on a capable local machine using the fast path.
  • Fall back to lower resolution or CPU simulation with a clear warning if WebGPU is unavailable or too slow.

Phased Implementation

Phase 1: Timeline And Correct Replay

  • Done: fix initial-stock simulation source.
  • Done: add simulation_timeline.v1 compiler.
  • Done: generate mid-operation stock keyframes for milling-style replay.
  • Done: export package assets without changing production geometry output.
  • Remaining: improve simulation/target tolerance validation and performance profiling on larger representative programs.

Phase 2: Interactive Viewer

  • Done: load simulation_timeline.json in the browser.
  • Done: add time-based scrub/playback using simulation keyframes.
  • Partial: animate tool pose using existing toolpath fallback when timeline segments do not contain enough pose data.
  • Remaining: render captions/tool-change/setup-transition overlays as explicit viewer UI.
  • Done: keep current static visualization behavior working.

Phase 3: Video Export

  • Add browser recording.
  • Add headless or CLI-driven video export.
  • Add deterministic camera presets and frame pacing.

Phase 4: Turning

  • Add lathe stock-profile simulation.
  • Render chuck, spindle rotation, turning tools, and progressive profile edits.

Phase 5: Process Coverage

  • Add V1 visual adapters for laser, press brake, EDM, grinder, and welding.
  • Label fidelity per process and per segment.
  • Promote any process from approximation to real geometry simulation only after tests prove the output.

Testing And Acceptance

Required tests:

  • Timeline compiler creates correct segments for pocket plus chamfer.
  • A mid-pocket timestamp shows partially removed material.
  • Final simulated stock matches final SubCAD output within tolerance.
  • Tool-change segments are inserted when the tool changes.
  • Machine/setup transition captions appear when process context changes.
  • Browser viewer loads and renders nonblank stock, tool, fixture, and captions.
  • Video export produces a playable file.
  • Existing simulation, toolpath, and visualization tests remain passing.

Performance acceptance:

  • Reference pocket/chamfer export reaches >=10x generation speed in fast mode.
  • CPU or reduced-resolution fallback reports its lower fidelity/performance.

Out Of Scope For V1

  • Production-certified G-code generation.
  • Full cutting-force, heat, chatter, deflection, or coolant physics.
  • Exact welding metallurgy or distortion.
  • Exact EDM plasma/spark material model.
  • Full press-brake springback prediction.
  • Full CAM strategy generation beyond the existing neutral toolpath intent.

Open Discussion Points

Before implementation, decide:

  • Whether review videos should default to real estimated time or a compressed review duration such as 10x playback.
  • Whether video export must support MP4 immediately or WebM is acceptable first.
  • What machine visuals should be modeled first: generic machines, named catalog machines, or both.
  • How much UI should live in the existing visualization viewer versus a new simulation-specific page.
  • Which process after milling and turning should be promoted from visual approximation to geometric simulation first.
